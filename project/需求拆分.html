    # 需求拆分

    • 需求拆分
    需求拆分是需求管理链路中的重要一环
    需求：创建 -> 沟通 -> 受理 -> 处理(拆分、分配) -> 评审 -> 开发 -> 测试 -> 发布 -> 验收

    一、需求拆分的价值：

    • 过程管理：降低需求交付风险
    • 质量管理：提升需求交付质量
    • 价值交付：缩短需求交付周期

    二、需求拆分的方法：
    ✅ (1)流程步骤拆分法
    ✅ (2)数据类别、范围或者边界拆分法
    ✅ (3)执行操作拆分方法
    ✅ (4)需求场景拆分方法
    ✅ (5)条件分支拆分方法
    ✅ (6)用户界面拆分方法
    ✅ (7)基本工作单元(最小价值单元)拆分方法

    三、需求拆分验收标准：
    结合INVEST原则，结合京东零售产研实际情况需同时具备以下五个特征：
    独立性(解除耦合依赖)：独立于其他需求，解除需求之间的依赖。
    有价值(PO认可价值)：PO即 产品经理的角色，作为用户的代言人需要认可拆分后的价值。
    可估算(工作量估算)：需求拆分后是可以估算工作量的。
    大小合适(拆分合适范围)：是将需求的规模拆分到大小合适范围，并不是说需求的越小越好。举例：如果需求的迭代周期是两周的话，那么开发+测试的工时(即：提供的大小合适范围)大概是2人/天至5人/天。如果需求迭代周期是三周的话，那么开+测试的工时(即：大小合适范围)大概是3人/天至7人/天。如果换算成小时，可以再乘以8。
    可测试(可独立测试)：可以独立测试。

    规模大、复杂、需要多人、多团队合作完成的需求要拆分

    四、团队成员对需求拆分如何分工：

    团队角色拆分职责：

    业务：

    • 1、基于业务场景和业务逻辑拆分；
    • 2、评审拆分后的交付物；

    PO(产品经理):

    • 1、负责需求拆分；
    • 2、组织团队校验拆分结果；
    • 3、将拆分后的需求处理至团队空间；

    研+测：

    • 1、预估需求工作量；
    • 2、参与拆分合理性校验；
    • 3、对于不满足标准的需求提供拆分方案；

    Scrum Master

    • 1、监测拆分结果，组织团队持续改进；
    • 2、提供拆分方法，指导团队拆分；
